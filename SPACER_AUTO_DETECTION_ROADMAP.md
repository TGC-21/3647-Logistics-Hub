# Onshape Auto-Detection Roadmap

## Goal

Build a reusable Onshape auto-detection pipeline that can identify internally generated fabrication candidates during assembly import, log the detection results into SQL, and let the user confirm fabrication parameters before a job is created.

The first detector will target spacer parts generated by the existing Spacer FeatureScript. The existing FeatureScript appears to have a unique enough signature for a first implementation, so modifying the FeatureScript is helpful but not required for spacer detection v1. The design should leave room for future detectors, such as a Shaft FeatureScript detector or other machineable part generators.

## Product Direction

Auto-detection should not silently send parts to fabrication.

Instead:

1. Partshelf imports the assembly BOM as it does today.
2. Behind the scenes, Partshelf detects likely generated fabrication parts.
3. Detection results are stored on the imported `assembly_parts` rows.
4. The Designer UI marks detected rows.
5. The user opens a confirmation overlay, reviews detected parameters, and confirms sending the part to fabrication.
6. Confirmed jobs enter the existing unbatched fabrication queue.

Completed fabrication jobs should continue using the existing Partshelf workflow: once complete, fabricated units become inventory instances.

## Detection Architecture

### Detector Registry

Create a generic detector system instead of hard-coding spacer logic directly into the BOM importer.

Suggested detector interface:

```js
{
  kind: 'spacer',
  generatorId: 'partshelf.spacer',
  candidateFilter(row) {},
  inspectPartStudio(context) {},
  matchBomRow(row, inspection) {},
  buildFabricationDraft(match) {}
}
```

Future detectors can follow the same pattern:

```js
{
  kind: 'shaft',
  generatorId: 'partshelf.shaft',
  ...
}
```

The importer should know only that detectors return fabrication drafts. It should not know spacer-specific formulas.

### Detection Layers

Use a layered approach:

1. **Cheap BOM prefilter**
   Use BOM row text and metadata to avoid unnecessary Onshape API calls.

2. **FeatureScript generator marker**
   Prefer stable generator metadata such as `PARTSHELF_GENERATOR_ID == "partshelf.spacer"` when available.

3. **Legacy FeatureScript signature**
   For old spacers, detect the known feature type, namespace, and parameter set.

4. **FeatureScript evaluation**
   Use `evalFeatureScript` only when the feature-list response cannot provide final fabrication values.

5. **Geometry fallback**
   Keep this as a future manual-review path only. It should not auto-queue fabrication.

## Spacer FeatureScript Contract

### Existing Spacer Values

The current Spacer FeatureScript already contains the values needed to fabricate a spacer:

- `definition.externalDia` is the spacer outer diameter.
- `definition.myLength` is the spacer length for blind spacers.
- For `UP_TO_FACE`, the script computes `definition.myLength` from `evDistance`.
- `definition.origin` is the selected circular edge used to derive the internal hole.
- `definition.internalAdd` is added to the selected circular edge diameter.

The fabrication dimensions are:

```text
OD = definition.externalDia
ID = 2 * originCirc.radius + definition.internalAdd
Length = definition.myLength
```

Body names must remain human-readable only. Users may rename spacer bodies to names like `0500-Subassembly plate spacer`, so names should not be used as machine detection signals.

### Add Stable Generator Constants

Recommended, but not required for spacer detection v1: add constants near the top of the Spacer FeatureScript:

```fs
const PARTSHELF_GENERATOR_ID = "partshelf.spacer";
const PARTSHELF_GENERATOR_VERSION = "1";
```

These values become the long-term detection contract for future versions. Avoid using display names, body names, or feature names as detection signals because users can edit them.

For spacer v1, the existing FeatureScript signature is probably unique enough:

- known FeatureScript namespace,
- `featureType == "myFeature"`,
- required parameters include `spacerType`, `internalAdd`, `externalDia`, `endType`, `myLength`, and `origin`.

This means the first implementation can detect legacy spacer features without requiring every existing Onshape document to be regenerated with new FeatureScript metadata.

### Add Computed Feature Metadata

Recommended for future robustness: after `originCirc` is evaluated and after `UP_TO_FACE` has resolved `definition.myLength`, compute:

```fs
var spacerId = 2 * originCirc.radius + definition.internalAdd;
var spacerOd = definition.externalDia;
var spacerLength = definition.myLength;
```

Then expose machine-readable metadata:

```fs
setFeatureComputedParameter(context, id, {
    "name" : "PARTSHELF_GENERATOR_ID",
    "value" : PARTSHELF_GENERATOR_ID
});

setFeatureComputedParameter(context, id, {
    "name" : "PARTSHELF_GENERATOR_VERSION",
    "value" : PARTSHELF_GENERATOR_VERSION
});

setFeatureComputedParameter(context, id, {
    "name" : "_partshelf_kind",
    "value" : "spacer"
});

setFeatureComputedParameter(context, id, {
    "name" : "_partshelf_id",
    "value" : spacerId
});

setFeatureComputedParameter(context, id, {
    "name" : "_partshelf_od",
    "value" : spacerOd
});

setFeatureComputedParameter(context, id, {
    "name" : "_partshelf_length",
    "value" : spacerLength
});
```

Proof-of-concept note: an old `getPartStudioFeatures` response did not expose resolved computed `_name`; it showed the feature name as `"#_name"`. So the implementation should test whether new computed parameters appear in the feature-list response. If not, use `evalFeatureScript` to retrieve final dimensions when needed.

## Request-Conserving Import Strategy

The API budget is limited, so detection should be source-based, not row-based.

### 1. Prefilter BOM Rows

During BOM parsing, consider rows as detector candidates only when cheap local signals match.

For the spacer detector:

- Include names or part numbers containing `spacer`.
- Optionally include `standoff` later if that becomes useful.
- Exclude likely COTS/vendor rows before making extra API calls.
- Exclude `itemSource.isStandardContent === true`.
- Exclude rows whose source document is not the same document as the imported assembly's root document.
- Exclude known vendor prefixes or manufacturer-owned rows when possible.

Project rule: if the spacer origin document does not belong to the same document the assembly comes from, treat it as COTS or external-source and do not auto-detect it for fabrication.

This name filter is only a request-saving prefilter. The final detection source of truth remains the FeatureScript marker or legacy signature.

### 2. Group Candidate Rows by Source Part Studio

Use `assembly_parts.onshape_reference.itemSource` from each candidate row:

```json
{
  "documentId": "...",
  "wvmType": "v",
  "wvmId": "...",
  "elementId": "...",
  "partId": "...",
  "partIdentity": "...",
  "sourceElementMicroversionId": "...",
  "configuration": "default",
  "fullConfiguration": "..."
}
```

Group candidates by:

```text
documentId + wvmType + wvmId + elementId + fullConfiguration
```

Then query each unique source Part Studio once.

### 3. Fetch Features Once Per Candidate Source

Use:

```text
GET /partstudios/d/{did}/{wvm}/{wvmid}/e/{eid}/features?rollbackBarIndex=-1&includeGeometryIds=true&noSketchGeometry=false
```

Manual test call:

```bash
curl -u "$ONSHAPE_ACCESS_KEY:$ONSHAPE_SECRET_KEY" \
  -H "Accept: application/json;charset=UTF-8; qs=0.09" \
  "https://cad.onshape.com/api/v9/partstudios/d/{documentId}/{wvmType}/{wvmId}/e/{elementId}/features?rollbackBarIndex=-1&includeGeometryIds=true&noSketchGeometry=false"
```

Use values from `onshape_reference.itemSource`.

### 4. Cache During Import

Cache feature-list and evaluation results by source Part Studio key for the duration of the import:

```text
documentId + wvmType + wvmId + elementId + fullConfiguration
```

Do not call Onshape again while rendering Designer or Fabricate. Store detection results in SQL.

## SQL Logging Plan

### Add Fabrication Detection Metadata

Add a JSON column to `assembly_parts`:

```sql
fabrication_metadata jsonb
```

Suggested shape:

```json
{
  "autoDetected": true,
  "kind": "spacer",
  "status": "detected",
  "confidence": "high",
  "source": "onshape-featurescript",
  "generator": "partshelf.spacer",
  "generatorVersion": "1",
  "dimensions": {
    "id": { "value": 0.196, "unit": "in" },
    "od": { "value": 0.375, "unit": "in" },
    "length": { "value": 1.25, "unit": "in" }
  },
  "overrides": null,
  "fabricationDraft": {
    "method": "Lathe",
    "quantityRequested": 2,
    "requiresConfirmation": true
  },
  "onshape": {
    "featureId": "...",
    "documentId": "...",
    "wvmType": "v",
    "wvmId": "...",
    "elementId": "...",
    "partId": "...",
    "partIdentity": "..."
  },
  "warnings": []
}
```

Recommended status values:

```text
none
detected
needs_review
confirmed
queued
ignored
failed
```

Use `detected` for high-confidence rows that still need user confirmation. Use `queued` only after the user confirms and a fabrication job is created.

Keep this separate from `onshape_reference`, which should remain the raw imported BOM row.

If the user edits confirmed parameters, store those edits as overrides while preserving the original detected values:

```json
{
  "dimensions": {
    "id": { "value": 0.196, "unit": "in" },
    "od": { "value": 0.375, "unit": "in" },
    "length": { "value": 1.25, "unit": "in" }
  },
  "overrides": {
    "length": {
      "value": 1.24,
      "unit": "in",
      "reason": "User confirmation edit"
    }
  }
}
```

### Optional Future Table

If detection history becomes useful, add a separate table later:

```sql
fabrication_detections (
  id text primary key,
  assembly_part_id text references assembly_parts(id) on delete cascade,
  kind text not null,
  status text not null,
  confidence text not null,
  metadata jsonb not null default '{}',
  created_at timestamptz not null default now()
)
```

For the first version, the JSON column on `assembly_parts` is simpler and enough.

## Backend Implementation Phases

### Phase 1: Detection Schema Only

Add `assembly_parts.fabrication_metadata`.

No UI changes yet. No fabrication jobs created automatically.

### Phase 2: Detector Registry

Add server-side detector modules:

```text
api/_lib/fabrication-detectors.js
api/_lib/detectors/spacer.js
```

The registry should:

1. Run each detector's cheap `candidateFilter` against BOM rows.
2. Group candidate rows by source Part Studio.
3. Fetch each candidate Part Studio once.
4. Ask each detector to inspect features and return matches.
5. Attach fabrication metadata to the rows inserted into `assembly_parts`.

### Phase 3: Spacer Detector

Implement the spacer detector:

1. Prefilter rows containing `spacer` in name or part number.
2. Exclude likely COTS or standard-content spacers.
3. Exclude spacer candidates whose source document differs from the imported assembly's root document.
4. Detect legacy spacer features by known namespace and parameter signature:
   - `featureType == "myFeature"`
   - `spacerType`
   - `internalAdd`
   - `externalDia`
   - `endType`
   - `myLength`
   - `origin`
5. Prefer new FeatureScript metadata with `PARTSHELF_GENERATOR_ID == "partshelf.spacer"` if it exists, but do not require it for v1.
6. Use `evalFeatureScript` only when necessary to resolve ID or up-to-face length.
7. Return a fabrication draft, not a fabrication job.

### Phase 4: Import Logging

Update `seedAssemblyContents` so each inserted `assembly_parts` row can include `fabrication_metadata`.

For this phase, verify in SQL that:

- spacer candidates are marked,
- non-spacers are not marked,
- COTS spacers are ignored or marked `ignored`,
- ambiguous rows are marked `needs_review`,
- no fabrication jobs are created.

### Phase 5: Reimport Behavior

On reimport:

1. Recompute detection metadata from current Onshape data.
2. Preserve existing behavior for inventory release.
3. Avoid creating jobs automatically.
4. If a part was previously confirmed and queued, decide whether to preserve the active job or ask the user to reconfirm when the part row is rebuilt.

For the first version, prefer reconfirmation if the old `assembly_parts` row is deleted and recreated.

## UI Implementation Phases

### Phase 6: Row Markers

In Designer assembly part rows, show a compact marker when `fabrication_metadata.autoDetected` is true.

Suggested marker states:

```text
Detected
Review
Queued
Ignored
```

The row marker should not imply the job already exists unless the status is `queued`.

### Phase 7: Confirmation Overlay

Add a button on detected rows that opens a fabrication confirmation overlay.

The overlay should show:

- part name,
- quantity needed,
- detected kind, such as `Spacer`,
- confidence,
- ID,
- OD,
- length,
- source FeatureScript/generator,
- warnings, if any,
- target fabrication method,
- quantity to queue.

Primary action:

```text
Confirm and Queue
```

Secondary actions:

```text
Edit parameters
Ignore
Cancel
```

On confirmation:

1. Create a `fabrication_jobs` row using the existing fabrication flow.
2. Leave the job unbatched.
3. Update `fabrication_metadata.status` to `queued`.
4. Store any user-edited parameters in `fabrication_metadata.overrides` while preserving the original detected values.

### Phase 8: Review Filter

Add Designer filters or quick views:

- detected fabrication candidates,
- needs review,
- queued for fabrication.

This helps users find all auto-detected rows after a large assembly import.

### Phase 9: Fabricate View Integration

In Fabricate mode, show confirmed detected parameters on queued jobs so machinists can see the source dimensions without returning to Designer.

This should read from the linked `assembly_parts.fabrication_metadata`.

## Confidence Rules

High confidence:

- The detector confirms `PARTSHELF_GENERATOR_ID == "partshelf.spacer"` or another known generator id.
- ID, OD, and length are all present and unit-bearing.
- The BOM row maps to one generated part, or the source Part Studio has only one detected generated part.

Medium confidence:

- Legacy spacer feature signature is present.
- Values can be reconstructed from raw parameters or `evalFeatureScript`.
- Mapping to a specific part is probable but not exact.

Needs review:

- Name suggests spacer but no generator or legacy signature is found.
- Multiple generated features exist in the same Part Studio and row-to-feature matching is ambiguous.
- Unit conversion or source mapping is incomplete.

Ignored:

- Name suggests spacer but row is standard content, known vendor/COTS, or outside the allowed source rules.
- Source document differs from the imported assembly's root document.

Only high-confidence detections should offer one-click confirmation. Medium-confidence detections should open the same overlay with warnings.

## Testing Plan

Create or identify Onshape fixtures for:

1. Round spacer, blind length.
2. Round spacer, up-to-face length.
3. 1/2 inch hex spacer.
4. 3/8 inch hex spacer.
5. Assembly with repeated instances of the same spacer.
6. Assembly with multiple different spacers from the same Part Studio.
7. Assembly with non-spacer parts.
8. Assembly with COTS/vendor spacers that should not be selected.
9. Old spacer document without new generator metadata.
10. Future shaft FeatureScript fixture once that generator exists.

For each fixture, verify:

- BOM import still works.
- Detection metadata is written to SQL.
- Non-spacers are not marked.
- COTS spacers are ignored.
- ID, OD, and length match Onshape for spacer detections.
- No fabrication job is created before user confirmation.
- Confirmation creates exactly one active fabrication job.
- Confirmed jobs appear in the unbatched queue.
- Reimport does not silently duplicate queued jobs.

## Recommended Implementation Order

1. Add `fabrication_metadata` to `assembly_parts`.
2. Add a generic detector registry.
3. Add request-conserving candidate grouping by source Part Studio.
4. Add the spacer detector with name prefilter and COTS exclusions.
5. Add Part Studio feature-list helper.
6. Test old spacer FeatureScript signature detection.
7. Add `evalFeatureScript` fallback only where needed.
8. Optionally modify Spacer FeatureScript to expose `PARTSHELF_GENERATOR_ID` and dimensions for future robustness.
9. If modified, test whether computed parameters appear in `getPartStudioFeatures`.
10. Store detection results during import, with no UI changes yet.
11. Add Designer row markers.
12. Add confirmation overlay.
13. Wire confirmation to existing fabrication job creation.
14. Add review filters.
15. Add Fabricate view dimension display.
16. Generalize the detector registry with a second detector, such as shafts, when ready.

## Open Questions

- Does `itemSource.partId` or `itemSource.partIdentity` map cleanly to the body created by the Spacer FeatureScript? Current assumption: both identify parts within the Part Studio, and `partIdentity` is likely the stronger differentiator because it is longer.
- Does `getPartStudioFeatures` include computed parameters set with `setFeatureComputedParameter`, or will those require `evalFeatureScript`?
- Answered: if the spacer source document differs from the imported assembly's root document, treat it as COTS/external and exclude it from auto-detection.
- Answered: confirmed parameter edits should create overrides while preserving original auto-detected values.
